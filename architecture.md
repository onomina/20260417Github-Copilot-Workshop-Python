# ポモドーロタイマーWebアプリ アーキテクチャ案

## 1. ディレクトリ構成（推奨）

```
1.pomodoro/
├── app.py                # Flaskアプリ本体
├── timer.py              # タイマー・ポモドーロロジック（テスト容易化のため分離）
├── templates/
│   └── index.html        # メイン画面テンプレート
├── static/
│   ├── css/
│   │   └── style.css     # スタイルシート
│   └── js/
│       └── timer.js      # タイマー制御用JS
└── tests/
    ├── test_timer.py     # ロジックのユニットテスト
    └── test_app.py       # Flaskアプリ/APIのテスト
```

## 2. 機能分担

- **Flask（バックエンド）**
  - ルーティング（画面表示、APIエンドポイント）
  - 必要に応じてセッション管理やユーザー認証
  - タイマーや履歴の状態保存（必要ならサーバー側で管理）

- **HTML/CSS（フロントエンド）**
  - UIレイアウト（タイマー表示、開始/停止/リセットボタン等）
  - スタイリング（レスポンシブ対応も推奨）

- **JavaScript（フロントエンド）**
  - タイマーのカウントダウン制御
  - ボタン操作のイベント処理
  - 必要に応じてAPIとの通信（履歴保存や状態同期）

## 3. 状態管理

- タイマーの状態は基本的にフロントエンド（JS）で管理
- 履歴やユーザーごとの状態保存が必要な場合のみサーバー側で管理

## 4. API設計例

- `/` : メイン画面（テンプレートでレンダリング）
- `/api/start`, `/api/stop`, `/api/reset` : タイマー操作用API（必要に応じて）
- `/api/history` : ポモドーロ履歴取得・保存API（必要に応じて）

## 5. ユニットテスト容易化のための工夫

- タイマーやポモドーロ管理のロジックはFlaskルートから分離し、独立したPythonモジュール（timer.py）として実装
- テスト用ディレクトリ（tests/）を用意し、pytest等でロジックやAPIのテストを実装
- JavaScriptもロジック部分はクラスやモジュール化し、テストしやすい構造に
- 依存性注入や副作用の少ない設計を意識

---

このアーキテクチャをベースに開発を進めることで、保守性・拡張性・テスト容易性の高いWebアプリが実現できます。
